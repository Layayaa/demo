## 遇到的问题及解决

### 问题 1：权限不足（polkit / Permission denied）

- **阶段**：服务器环境配置
- **现象**：执行 `systemctl enable docker` 时报 `pam_authenticate failed: Authentication failure`；执行 `apt-get install` 时报 `Permission denied`
- **原因**：默认以非 root 用户 SSH 登录，执行系统级命令缺乏权限
- **解决**：`sudo su -` 切换到 root 用户后再执行所有命令

### 问题 2：登录页面显示"请求过于频繁，请稍后重试"

- **阶段**：首次部署完成后测试
- **现象**：登录接口返回 429 状态码
- **原因**：`/api/login` 限流规则为 5次/300秒（5分钟），多次尝试后被锁定
- **解决**：将限流放宽为 10次/60秒（`app.py` 第 87 行）

### 问题 3：管理员账号未创建，密码正确也无法登录

- **阶段**：首次部署
- **现象**：登录返回"手机号或密码错误"，但使用的是默认管理员密码
- **根本原因**：Gunicorn 启动时 `if __name__ == '__main__'` 代码块不执行，因此 `init_db()` → `create_initial_admin()` 从未被调用，数据库中没有任何用户
- **解决**：将初始化逻辑提取为 `startup_init()` 函数，放在模块顶层执行，确保无论是 `python app.py` 还是 `gunicorn app:app` 都能正确初始化

### 问题 4：密码哈希字段被截断（线上 MySQL）

- **阶段**：修复问题 3 后重新部署
- **现象**：管理员账号已创建，但登录仍显示"手机号或密码错误"
- **根本原因**：Werkzeug 3.x 默认使用 `scrypt` 算法，生成的哈希值长度约 160 字符，但 MySQL 中 `password_hash` 字段为 `VARCHAR(128)`，哈希被截断，导致 `check_password_hash()` 永远返回 False
- **解决**：
  1. `models.py` 中 `password_hash` 字段扩大为 `String(256)`
  2. `schema.sql` 中 `password_hash` 字段扩大为 `VARCHAR(256)`
  3. `docker compose down -v` 清除旧数据卷，重建数据库

### 问题 5：本地 Python 3.9 不支持 scrypt 算法

- **阶段**：本地开发环境测试
- **现象**：本地登录返回 500 错误，日志报 `AttributeError: module 'hashlib' has no attribute 'scrypt'`
- **根本原因**：macOS 系统自带 Python 3.9.6 的 `hashlib` 未编译 scrypt 支持，而 Werkzeug 3.x 的 `generate_password_hash()` 默认使用 scrypt
- **解决**：`models.py` 中 `set_password()` 显式指定 `method='pbkdf2:sha256'`，该算法在所有 Python 版本上都可用
- **附带操作**：删除本地 `database/user.db`，重启应用使其用新算法重建管理员密码

### 问题 6：多次修复后线上数据不一致

- **阶段**：多轮修复上传后
- **现象**：部分文件已更新但 MySQL 中仍有旧数据（旧哈希格式、旧字段长度）
- **解决**：最终采用完整重新部署方案——清除服务器旧文件和 Docker 数据卷，完整重新上传项目，从零构建

## 经验总结

| 教训 | 说明 |
|------|------|
| Gunicorn 不执行 `__main__` | 应用初始化逻辑（建表、建默认用户等）必须在模块加载时执行，不能放在 `if __name__ == '__main__'` 中 |
| 密码哈希字段要预留足够长度 | Werkzeug 不同版本的哈希算法输出长度不同，建议至少 `VARCHAR(256)` |
| 指定哈希算法确保兼容 | 不同 Python 版本对 hashlib 算法支持不同，生产代码应显式指定 `method='pbkdf2:sha256'` |
| 限流策略要合理 | 登录接口 5次/5分钟 太严格，调试阶段容易把自己锁住 |
| Docker 部署修改后要清卷重建 | MySQL 数据持久化在 volume 中，改了 schema 后必须 `docker compose down -v` 才能生效 |
